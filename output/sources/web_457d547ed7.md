# 🤖 OpenAI Symphony: AI 코딩 에이전트 오케스트레이터 완벽 가이드

안녕하세요! 오늘은 OpenAI가 공개한 Symphony에 대해 알아보겠습니다. Symphony는 이슈 트래커와 AI 코딩 에이전트를 연결하여 프로젝트 작업을 자동으로 처리하는 오픈소스 오케스트레이터입니다. 이 블로그에서는 Symphony의 핵심 기능과 하네스 엔지니어링 개념, 설치 방법, 대시보드 설정까지 상세히 다루어 보겠습니다. 바로 시작해볼까요?
"이 포스팅은 쿠팡 파트너스 활동의 일환으로, 이에 따른 일정액의 수수료를 제공받습니다."
🤖 Symphony란 무엇인가요?
Symphony는 OpenAI가 2026년 3월에 공개한 오픈소스 프레임워크로, AI 코딩 에이전트를 자동으로 관리하고 실행하는 오케스트레이터입니다. 쉽게 말해, 프로젝트 관리 도구(예: Linear)에 등록된 작업을 AI가 자동으로 처리하도록 돕는 시스템입니다.
기존에는 개발자가 AI 코딩 도구(Cursor, Claude Code, Codex 등)를 직접 실행하고 감시해야 했습니다. 하지만 Symphony를 사용하면 이슈가 등록되면 AI가 자동으로 작업을 시작하고, 완료되면 결과를 보고합니다. 개발자는 "무엇을 해야 하는지"만 관리하면 되고, "어떻게 하는지"는 AI에 맡길 수 있습니다.
핵심 개념 이해하기
- 오케스트레이터(Orchestrator): 여러 AI 에이전트를 조율하고 관리하는 중앙 시스템. 마치 지휘자가 오케스트라를 이끄는 것과 같습니다.
- 이슈 트래커(Issue Tracker): 프로젝트의 할 일을 관리하는 도구. Linear, Jira, GitHub Issues 등이 있습니다. Symphony는 현재 Linear를 기본 지원합니다.
- Linear: 실리콘밸리 스타트업들이 많이 사용하는 현대적인 프로젝트 관리 도구입니다. Jira보다 훨씬 간단하고 직관적인 인터페이스가 특징이며, 개발자 친화적인 단축키와 키보드 내비게이션을 제공합니다. GraphQL API를 통해 강력한 자동화가 가능합니다.
- 구현 실행(Implementation Run): 하나의 이슈를 해결하기 위한 AI 에이전트의 작업 단위. 각 실행은 격리된 환경에서 수행됩니다.
⚙️ Symphony 핵심 기능 살펴보기
Symphony는 단순한 자동화 도구가 아닙니다. AI 에이전트가 안정적으로 작업할 수 있도록 설계된 종합 관리 시스템입니다. 주요 기능을 살펴보겠습니다.
1. 이슈 폴링 및 자동 실행
Symphony는 Linear와 같은 이슈 트래커를 주기적으로 확인합니다. 설정된 상태(예: "Todo", "In Progress")의 이슈가 있으면 자동으로 AI 에이전트를 실행합니다. 개발자가 별도로 명령을 내릴 필요가 없습니다.
2. 격리된 워크스페이스
각 이슈는 별도의 격리된 디렉토리에서 처리됩니다. 예를 들어, ABC-123
이슈는 ~/symphony_workspaces/ABC-123/
폴더에서 작업합니다. 이렇게 하면 여러 이슈를 동시에 처리할 때 서로 충돌하지 않습니다.
3. 재시도 및 백오프 메커니즘
AI 작업이 실패해도 Symphony는 자동으로 재시도합니다. 지수 백오프(Exponential Backoff) 방식을 사용하여 점진적으로 재시도 간격을 늘립니다. 첫 번째 재시도는 1분 후, 두 번째는 2분 후, 세 번째는 4분 후 식입니다. 최대 대기 시간은 5분으로 설정됩니다.
4. 상태 조정(Reconciliation)
이슈의 상태가 변경되면 Symphony가 이를 감지하고 적절히 대응합니다. 예를 들어, AI가 작업 중인 이슈가 "Done" 상태로 바뀌면 작업을 중단하고 워크스페이스를 정리합니다.
5. HTTP 대시보드
실행 중인 이슈, 재시도 큐, 토큰 사용량 등을 실시간으로 모니터링할 수 있는 웹 대시보드를 제공합니다. 브라우저에서 http://localhost:8080
으로 접속하면 현재 상태를 한눈에 볼 수 있습니다.
| 기능 | 설명 | 작동 방식 |
|---|---|---|
| 이슈 폴링 | 이슈 트래커 주기적 확인 | 30초마다 Linear API 호출 |
| 워크스페이스 격리 | 이슈별 독립 디렉토리 | ~/symphony_workspaces/{이슈ID}/ |
| 재시도 큐 | 실패 시 자동 재시도 | 지수 백오프 (최대 5분) |
| 상태 조정 | 이슈 상태 변경 감지 | 실행 중단/워크스페이스 정리 |
| HTTP 대시보드 | 실시간 모니터링 | 포트 8080에서 웹 서버 실행 |
🏗️ 시스템 아키텍처 이해하기
Symphony는 Elixir와 Erlang/BEAM 런타임으로 구축되었습니다. 왜 굳이 Python이나 TypeScript가 아닌 Elixir를 선택했을까요? AI 에이전트 작업은 장시간 실행되며 실패 가능성이 있습니다. Elixir의 슈퍼비전 트리(Supervision Tree)는 수백 개의 독립적인 작업을 동시에 관리하면서, 하나가 실패해도 전체 시스템에 영향을 주지 않습니다. 이는 AI 자동화에 매우 적합한 구조입니다.
💡 참고: 슈퍼비전 트리란?
Erlang/BEAM의 핵심 개념으로, 각 프로세스를 감시하다가 실패 시 자동으로 복구하는 구조입니다. "Let it crash(충돌해도 괜찮다)" 철학을 기반으로, 에러를 방지하는 대신 빠르게 복구하는 데 집중합니다.
Symphony는 또한 WORKFLOW.md 파일을 통해 설정을 관리합니다. 이 파일은 저장소에 포함되어 버전 관리되므로, AI의 행동 규칙이 코드와 함께 관리됩니다.
📋 사전 요구사항 확인: Harness Engineering
Symphony를 효과적으로 사용하려면 프로젝트가 Harness Engineering 원칙을 따라야 합니다. OpenAI는 2026년 2월, 이 방법론을 공식적으로 소개하며 "에이전트 우선 세계에서 Codex 활용하기"라는 제목의 기술 문서를 발표했습니다. 이는 AI가 코드를 이해하고 수정하기 쉽도록 프로젝트를 구조화하는 방법론입니다.
아래 개념도는 하네스 엔지니어링에 대한 인포그래픽입니다.(구글 NotebookLM 생성)
Harness Engineering이란?
Harness Engineering은 OpenAI가 5개월간 실험을 통해 검증한 방법론입니다. 핵심은 "사람은 조정하고, 에이전트는 수행한다"는 것입니다. OpenAI 팀은 실제로 수동으로 코드를 한 줄도 작성하지 않고 백만 라인의 코드를 가진 소프트웨어 제품을 구축했습니다. 이 과정에서 엔지니어의 역할이 코드 작성에서 환경 설계, 의도 명시, 피드백 루프 구축으로 변화했습니다.
Harness Engineering 핵심 요구사항
- 밀폐된 테스트(Hermetic Testing): 외부 의존성 없이 로컬에서 실행 가능한 테스트. CI/CD 파이프라인에서 독립적으로 실행될 수 있어야 합니다.
- 기계 판독 가능 문서: AI가 빌드, 테스트, 배포 방법을 스스로 파악할 수 있는 문서.
AGENTS.md
,ARCHITECTURE.md
등의 표준화된 문서가 필요합니다. - 모듈형 아키텍처: 부작용이 최소화된 코드 구조. 명확한 계층 구조와 의존성 방향이 정의되어야 합니다.
- 명시적인 경계: 각 모듈이나 도메인의 책임과 인터페이스가 명확히 정의되어야 합니다.
왜 Harness Engineering이 필요할까요?
OpenAI의 실험에서 밝혀진 핵심 통찰은 "에이전트가 볼 수 없는 것은 존재하지 않는 것"이라는 점입니다. Google Docs, Slack 대화, 사람의 머릿속에 있는 지식은 AI 에이전트가 접근할 수 없습니다. 오직 리포지터리 내의 버전 관리되는 아티팩트(코드, 마크다운, 스키마 등)만이 AI가 인식할 수 있습니다.
💡 실제 사례: OpenAI의 5개월 실험
- 코드 규모: 약 백만 라인의 코드를 5개월 만에 생성
- 팀 규모: 초기 3명, 현재 7명의 엔지니어
- PR 처리량: 엔지니어 1인당 하루 평균 3.5개 PR
- 생산성 향상: 수동 작성 대비 약 10배 빠른 개발 속도
문서화 전략: AGENTS.md 활용법
Harness Engineering에서 가장 중요한 것은 지식 베이스의 체계적인 관리입니다. OpenAI는 거대한 단일 매뉴얼 대신 목차 역할을 하는 짧은 AGENTS.md(약 100줄)를 권장합니다:
리포지터리 구조 예시:
AGENTS.md # 목차 역할 (약 100줄)
ARCHITECTURE.md # 아키텍처 개요
docs/
├── design-docs/ # 설계 문서
├── exec-plans/ # 실행 계획
├── product-specs/ # 제품 사양
├── references/ # 참조 문서
├── DESIGN.md # 설계 원칙
├── QUALITY.md # 품질 기준
└── SECURITY.md # 보안 가이드라인
아키텍처 일관성 유지하기
OpenAI는 레이어드 도메인 아키텍처를 사용하여 AI 에이전트가 실수하지 않도록 강제합니다. 각 비즈니스 도메인은 고정된 계층 구조를 따르며, 이를 맞춤형 린터와 구조적 테스트로 자동 검증합니다:
| 계층 | 역할 | 의존성 규칙 |
|---|---|---|
| Types | 타입 정의 | 다른 계층에 의존하지 않음 |
| Config | 설정 관리 | Types만 의존 |
| Repo | 데이터 접근 | Types, Config 의존 |
| Service | 비즈니스 로직 | Repo, Providers 의존 |
| Runtime | 실행 환경 | Service 의존 |
| UI | 사용자 인터페이스 | Runtime 의존 |
핵심 철학: "수동으로 작성한 코드가 없다"
OpenAI 팀의 가장 흥미로운 실험 규칙은 "No manually written code"였습니다. 모든 코드를 Codex가 작성하도록 강제함으로써, 엔지니어들은 코드 작성이 아닌 시스템 설계와 피드백 루프 구축에 집중할 수 있었습니다. 결과적으로:
- 코드 리뷰: 대부분 에이전트 간에 처리됨
- 버그 수정: 에이전트가 직접 재현, 수정, 검증
- 문서화: 모든 의사결정이 리포지터리에 기록됨
- 품질 관리: 맞춤형 린터로 일관성 자동 유지
⚠️ 주의사항
Harness Engineering은 초기 설정 비용이 필요합니다. 프로젝트 구조 재설계, 문서화 표준 수립, 자동화 도구 구축에 시간을 투자해야 합니다. 하지만 일단 설정되면 AI 에이전트의 생산성이 비약적으로 향상됩니다.
이러한 준비가 되어 있지 않으면 AI가 작업 결과를 검증하기 어렵습니다. Symphony를 도입하기 전에 프로젝트 구조를 점검해 보세요.
🚀 설치 및 설정 방법
Symphony는 두 가지 방법으로 설치할 수 있습니다. 공식 Elixir 구현체를 사용하거나, 직접 원하는 언어로 구현하는 것입니다.
방법 1: 직접 구현하기 (권장)
OpenAI는 언어에 구애받지 않는 사양서(SPEC.md)를 제공합니다. 원하는 프로그래밍 언어로 직접 구현할 수 있습니다. TypeScript, Python, Go 등 어떤 언어든 가능합니다. 아래와 같은 내용을 원하는 Claude Code같은 AI 에이전트에게 요청하면 됩니다.
# AI 에이전트에게 요청:
"Implement Symphony according to the following spec:
https://github.com/openai/symphony/blob/main/SPEC.md"
방법 2: 공식 Elixir 구현체 사용
공식 저장소의 elixir/
디렉토리에서 참조 구현체를 확인할 수 있습니다.
# 저장소 클론
git clone https://github.com/openai/symphony.git
cd symphony/elixir
# 의존성 설치
mix deps.get
# 환경 설정
export LINEAR_API_KEY="your-api-key-here"
# 실행
mix run --no-halt
📝 WORKFLOW.md 설정 가이드
WORKFLOW.md는 Symphony의 핵심 설정 파일입니다. YAML 프론트매터와 마크다운 프롬프트로 구성됩니다.
---
tracker:
kind: linear
api_key: $LINEAR_API_KEY # 환경 변수 사용
project_slug: YOUR-PROJECT-ID # Linear 프로젝트 ID
active_states:
- Todo
- In Progress
terminal_states:
- Done
- Closed
- Cancelled
polling:
interval_ms: 30000 # 30초마다 폴링
workspace:
root: ~/symphony_workspaces # 워크스페이스 루트
hooks:
after_create: |
git clone https://github.com/your-org/your-repo.git .
npm install
before_run: |
git pull origin main
agent:
max_concurrent_agents: 10
max_turns: 20
server:
port: 8080 # HTTP 대시보드 포트
---
You are a software development agent working on issues from Linear.
## Issue Information
- **Identifier**: {{ issue.identifier }}
- **Title**: {{ issue.title }}
- **Description**: {{ issue.description }}
## Guidelines
- Write clean, maintainable code
- Test your changes thoroughly
- Create a clean commit with a descriptive message
주요 설정 항목 설명
| 설정 항목 | 설명 | 기본값 |
|---|---|---|
| tracker.api_key | Linear API 키 ($ENV_VAR 형식 지원) | 필수 |
| tracker.project_slug | Linear 프로젝트 ID | 필수 |
| tracker.active_states | 작업 대상 상태 목록 | ["Todo", "In Progress"] |
| polling.interval_ms | 폴링 간격 (밀리초) | 30000 (30초) |
| workspace.root | 워크스페이스 저장 경로 | 시스템 임시 디렉토리 |
| server.port | HTTP 대시보드 포트 | 비활성화 |
Linear 프로젝트 설정 후 API 키는 아래 사이트에서 발급받을수 있습니다.
📊 HTTP 대시보드 확인 방법
Symphony는 선택적으로 HTTP 서버를 실행하여 실시간 모니터링을 제공합니다. 대시보드를 활성화하려면 WORKFLOW.md에 server.port
설정을 추가하세요.
server:
port: 8080
"bun run index.ts"로 Symphony를 실행한 후 브라우저에서 http://localhost:8080
으로 접속하면 아래와 같이 대시보드를 볼 수 있습니다.
대시보드에서 확인할 수 있는 정보
- 실행 중인 이슈: 현재 AI가 작업 중인 이슈 목록, 세션 ID, 시작 시간
- 재시도 큐: 실패하여 재시도 대기 중인 이슈와 예정 시간
- 토큰 사용량: 총 입력/출력 토큰 수
- 실행 시간: Symphony가 실행된 총 시간
REST API 엔드포인트
대시보드는 HTML뿐만 아니라 JSON API도 제공합니다.
| 엔드포인트 | 메서드 | 설명 |
|---|---|---|
| / | GET | HTML 대시보드 페이지 |
| /api/v1/state | GET | 현재 상태 JSON 응답 |
| /api/v1/{identifier} | GET | 특정 이슈 상태 조회 |
| /api/v1/refresh | POST | 즉시 폴링 트리거 |
⚠️ 주의사항
대시보드는 기본적으로 127.0.0.1
에서만 접근 가능합니다. 외부에서 접속하려면 별도의 리버스 프록시(nginx, Cloudflared 등)를 설정해야 합니다.
🔄 구현 실행 수명 주기
Symphony에서 하나의 이슈가 처리되는 과정은 다음과 같습니다.
- 폴링 및 트리거: 설정된 상태(예: "Todo")의 이슈를 감지
- 샌드박스 격리: 이슈 전용 워크스페이스 디렉토리 생성
- 에이전트 실행: AI가 이슈 내용을 읽고 코드 작성
- 작업 증명(Proof of Work): CI 통과, 테스트 성공 등 결과 검증
- 랜딩(Landing): PR 생성 또는 머지 수행
이 과정에서 실패가 발생하면 재시도 큐에 들어가고, 지수 백오프 후 다시 시도합니다.
💡 실제 활용 사례
Symphony는 어떤 상황에서 유용할까요?
추천 활용 시나리오
- 버그 수정 자동화: "Bug" 라벨이 붙은 이슈를 AI가 자동으로 분석하고 수정
- 기능 구현: 명확한 요구사항이 있는 작은 기능을 AI가 구현
- 리팩토링: 코드 정리, 파일 이동, 이름 변경 등 반복 작업
- 문서 업데이트: README, API 문서 등의 자동 갱신
이 기준으로 보면, 본인의 사용 패턴이 어느 쪽에 더 가까운지에 따라 선택이 달라질 수 있습니다.
⚠️ 한계점과 주의사항
Symphony는 강력한 도구이지만 몇 가지 한계가 있습니다.
- 신뢰 환경 필요: OpenAI는 이를 "trusted environments"용 엔지니어링 프리뷰라고 명시했습니다. 아직 프로덕션용이 아닙니다.
- Linear 의존성: 현재는 Linear만 공식 지원합니다. 다른 이슈 트래커는 직접 구현해야 합니다.
- Codex 필요: AI 에이전트로 OpenAI Codex를 사용합니다. Codex CLI가 설치되어 있어야 합니다.
- Harness Engineering 전제: 프로젝트가 기계 친화적이어야 합니다. 준비되지 않은 프로젝트에서는 효과가 떨어집니다.
🎯 맺음말
지금까지 OpenAI Symphony에 대해 알아보았습니다. Symphony는 이슈 트래커 연동, 격리된 워크스페이스, 자동화된 에이전트 실행, HTTP 대시보드 등 AI 코딩 자동화를 위한 핵심 기능을 제공하는 오픈소스 오케스트레이터입니다. 특히 Elixir 기반의 장애 허용 아키텍처로 수백 개의 작업을 안정적으로 관리할 수 있습니다.
다만 아직 엔지니어링 프리뷰 단계이며, 프로젝트가 Harness Engineering 원칙을 따라야 효과적입니다. 도입 전에 프로젝트 구조를 점검해 보시길 추천드립니다.
