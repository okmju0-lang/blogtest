# Thought Leadership 글 템플릿 (수요일: 문제 + 실행)

## 개요

- **전략적 역할**: 공감과 필요성 일깨우기 — "우리도 이 문제 있는데..." 반응 유도
- **핵심 차별화**: 문제만 쓰면 끝나고, 실행만 쓰면 관심 없다. 반드시 같이 간다.
- **톤**: 권위 있되 읽기 쉽게. 신뢰할 수 있는 전문가가 "이렇게 하면 됩니다"라고 말해주는 느낌
- **분량**: 2,000~3,000자

## 콘텐츠 구조 원칙

모든 변형에 공통 적용되는 4단 구조:

1. **문제 정의** — 독자가 "맞아, 우리도 이 문제 있어"라고 느끼게
2. **왜 발생하는가** — 구조적 원인 분석. 표면이 아닌 본질
3. **해결 방향** (핵심) — 명확한 방향 제시. 이것이 글의 중심
4. **간단한 실행 방법** — 바로 시도할 수 있는 구체적 액션

> 문제 분석에서 끝나는 글이 아니다. 반드시 "그래서 어떻게 하라는 건데?"에 답한다.

## 비유·실무 예시·제품 연계 규칙

- **일상 비유**: 기술/개념 설명 시 비전문가도 이해할 수 있는 비유를 적극 활용한다.
- **실무 예시**: 추상적 주장 직후 반드시 구체적 실무 상황을 덧붙인다.
- **제품 라인업 연계**: 글 주제와 직접 관련된 매직에꼴 제품이 있으면, 실행 방법 섹션에서 **한 줄로** 자연스럽게 언급한다. `references/ax-resources.md` 참조. 관련 제품이 없으면 생략.

## 디자인 요소

- **컬러 키**: 딥 블루 + 골드 (문제: 어두운 톤 → 해결: 밝은 골드로 전환)
- **썸네일**: 문제와 해결의 대비가 느껴지는 비주얼
- **문제 섹션**: 진한 톤으로 위기감/긴장감 전달
- **해결 섹션**: 밝은 톤으로 전환 — 인용 블록(`>`)으로 핵심 전환점 강조
- **실행 섹션**: 단계별 or 체크리스트 형태로 명확하게
- **제목 패턴**: 문제 키워드 + 해결 키워드 병치 ("~하는 이유 + 해결 방법")

## 구조 변형 선택 가이드

글마다 다른 느낌을 주기 위해, 소스 성격에 맞는 구조를 선택한다. **매번 같은 구조를 반복하지 않는다.**

| 변형 | 적합한 소스 | 느낌 |
|---|---|---|
| **A. 진단+처방형** (기본) | 데이터/리서치 기반, 산업 이슈 | "정확히 짚었다" — 논리적 설득 |
| **B. 스토리+전환형** | 실패/성공 사례, 변화 과정 | "우리 이야기 같다" — 감정적 공감 |
| **C. 프레임워크형** | 방법론, 전략 모델, 체계적 접근 | "이대로 하면 되겠다" — 실행 확신 |

## 작성 주의사항

- **문제+실행 균형 필수**: 문제 분석에서 끝나지 않는다. 반드시 구체적 실행 방법을 제시한다. 분량 배분: 문제(30%) + 원인(20%) + 해결(30%) + 실행(20%).
- **공감 우선**: 리드 문단에서 독자의 고민/상황에서 시작한다. 기술이나 이론에서 시작하지 않는다.
- **본문 포맷 유지**: 첫 번째를 제외한 주요 `##` 섹션 앞에는 `---`를 넣고, CTA 바로 앞에는 `---`를 넣지 않는다.
- **가독성 우선**: 문단 사이는 반드시 한 줄 이상 비우고, 목록/인용문/표 앞뒤에도 빈 줄을 둔다.
- 회사의 관점을 명확히 드러내되, 근거 없는 주장을 하지 않는다.
- 소스에 없는 통계나 연구 결과를 임의로 인용하지 않는다.
- 리드 문단에서 바로 본론으로 들어간다. 장황한 배경 설명 금지.
- **직전 포스트와 같은 구조 변형을 연속 사용하지 않는다.** 다양성이 핵심.
