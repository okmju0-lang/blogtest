# AI Trend 글 템플릿 (월요일: 신기술 트렌드)

## 개요

- **전략적 역할**: 리드 유입 — "내 문제 같다" 반응 유도 → 클릭 + 체류 확보
- **핵심 차별화**: 기술 설명(X) → **실무 문제 + 해결** 관점(O). 비전문가도 "내 상황"으로 느끼게
- **톤**: 편안하고 친근한 어조. 문제를 겪어본 사람이 해결법을 알려주는 느낌
- **분량**: 2,000~3,000자

## 콘텐츠 구조 원칙

모든 변형에 공통 적용되는 6단 구조:

1. **공감** — 실무 상황으로 시작. 독자가 "내 문제다"라고 느끼게
2. **문제 정의** — 기존 방식이 왜 안 되는지, 구체적 증상 나열
3. **사례** (핵심) — 실제 업무 상황에서 문제가 어떻게 드러나는지 보여주기
4. **해결** — 새로운 기술이 이 문제를 어떻게 푸는지. 직관적으로 차이가 느껴지게
5. **간단한 테크 설명** — "아 이런 구조구나" 정도만. 깊은 기술 설명은 배제
6. **언제 써야 하는지** — 상황별 선택 가이드로 마무리

> 기술을 설명하는 글이 아니다. **독자의 실무 문제에서 시작해서, 해결책으로 자연스럽게 기술을 소개하는 글**이다.

## 비유·실무 예시·제품 연계 규칙

새 개념을 소개할 때는 **관심 있는 비전문가도 끝까지 읽을 수 있어야** 한다.

- **일상 비유**: 모든 기술 개념에 최소 1개의 일상 비유를 붙인다. "도서관 사서 vs 학자", "엑셀 VLOOKUP vs 피벗 테이블" 같은 수준.
- **실무 예시**: 추상적 설명 직후 반드시 "예를 들어" 실무 상황을 덧붙인다. 독자의 업무 현장이 떠올라야 한다.
- **전문 용어 즉시 해소**: 전문 용어가 나오면 같은 문장 또는 바로 다음 문장에서 비유/예시로 풀어준다.
- **제품 라인업 연계**: 글 주제와 직접 관련된 매직에꼴 제품이 있으면, 해결 또는 선택 가이드 섹션에서 **한 줄로** 자연스럽게 언급한다. `references/ax-resources.md`의 "제품/서비스 라인업" 참조. 관련 제품이 없으면 생략.

## 디자인 요소

- **컬러 키**: 네온 블루 + 퍼플 그라디언트 (혁신, 미래)
- **썸네일**: 미래지향적, 역동적 비주얼
- **비교표**: 기존 방식 vs 새로운 방식 — 모든 변형에서 필수
- **기업 활용 박스**: 인용 블록(`>`)으로 실무 적용 시나리오 강조
- **제목 패턴**: **실무 문제 + 해결** 형태 ("~가 자꾸 틀리는 이유", "~했는데 이상하다면", "~가 안 나오는 이유와 해결 방법")

## 구조 변형 선택 가이드

글마다 다른 느낌을 주기 위해, 소스 성격에 맞는 구조를 선택한다. **매번 같은 구조를 반복하지 않는다.**

| 변형 | 적합한 소스 | 느낌 |
|---|---|---|
| **A. 문제 해결형** (기본) | 기술 발표, 연구 결과, 기존 기술의 한계를 넘는 새 기술 | "내 문제에 답이 있다" — 공감 → 해결 |
| **B. 실전 적용형** | 튜토리얼, 도구 활용법, 워크플로우 시연 | "이렇게 하면 된다" — 단계별 가이드 |
| **C. 관점 전환형** | 논쟁적 주제, 통념을 뒤집는 트렌드 | "알고 있던 것이 틀렸다" — 반전 + 해결 |

## 작성 주의사항

- **실무 문제에서 시작 필수**: 모든 변형에서 기술 설명이 아니라 **독자의 실무 문제/상황에서 글을 시작**한다. "이 기술이 뭔가"가 아니라 "이 문제를 겪고 있다면"으로 접근한다.
- **사례 섹션이 핵심**: 추상적 설명보다 구체적 업무 사례로 "보여주는" 것이 가장 중요하다. 설명하지 말고 보여준다.
- **테크 설명은 최소한으로**: 기술 원리는 "아 이런 구조구나" 수준으로만. 비전문가가 초반에 이탈하지 않도록 깊은 기술 설명을 배제한다.
- **비교표 필수**: 기존 방식 vs 새로운 접근을 명확히 보여주는 표를 반드시 포함한다.
- **소스 기반 엄수**: 소스에 없는 AI 모델 성능 수치, 연구 결과를 임의로 추가하지 않는다.
- **본문 포맷 유지**: 첫 번째를 제외한 주요 `##` 섹션 앞에는 `---`를 넣고, CTA 바로 앞에는 `---`를 넣지 않는다.
- **가독성 우선**: 문단 사이는 반드시 한 줄 이상 비우고, 목록/인용문/표 앞뒤에도 빈 줄을 둔다.
- 기술 용어는 처음 등장 시 괄호로 간략히 설명. 독자에 비기술 임원도 포함.
- **제목은 반드시 실무 문제 + 해결 형태**: "RAG란 무엇인가"(X) → "사내 챗봇이 자꾸 틀리는 이유"(O). 비전문가도 "내 문제 같다"고 클릭할 수 있어야 한다.
- 리드 문단에서 바로 핵심으로 들어간다. 장황한 서론 금지.
- **직전 포스트와 같은 구조 변형을 연속 사용하지 않는다.** 다양성이 핵심.
